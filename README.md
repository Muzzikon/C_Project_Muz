# C_Project_Muz

## О проекте

`C_Project_Muz` — учебный проект на языке **C** по теме **пространственных структур данных**, в центре которого находится **3D k-d tree** для работы с облаками точек из CSV.

Проект ориентирован на задачи, близкие к робототехнике и обработке пространственных данных:
- загрузка 3D-точек из CSV;
- построение 3D k-d tree;
- поиск ближайшего соседа;
- диапазонный поиск;
- сравнение `KD-Tree` и `Brute force` по корректности и времени;
- кластеризация `DBSCAN`.

На текущем этапе в проекте **не реализован `Fuzzy C-means`**. Режим `-cmeans` распознаётся в CLI, но пока оставлен как заглушка.

---

## Что уже реализовано

В проекте уже есть:

- загрузка точек из CSV в память;
- полноценное **3D KD-дерево**;
- режимы:
  - `-kd_insert`
  - `-kd_nearest`
  - `-kd_range`
  - `-dbscan`
- проверка корректности `nearest` и `range` через **brute force**;
- сравнение времени работы `KD-Tree` и `Brute force`;
- `DBSCAN` для 3D-точек;
- ускорение `DBSCAN` через **пространственную сетку (`grid index`)**, а не через `KD-дерево`;
- сохранение результатов `range query` и `DBSCAN` в отдельные CSV-файлы;
- 2D-визуализация результата `DBSCAN` через Python-скрипт.

## История проекта в двух словах

Работа над проектом шла по этапам.

### 1. Сначала был 2D-этап

На раннем этапе были доведены до рабочего состояния:
- `nearest neighbor`;
- `range query`;
- сравнение с `brute force`;
- сравнение по времени.

Тогда же были добавлены вспомогательные функции:
- отдельный разбор точек из CLI;
- загрузка точек из CSV;
- сортировка и сравнение результатов диапазонного поиска.

### 2. Затем проект был переведён на 3D

После этого структура `Point` и всё ядро проекта были переведены с 2D на 3D:
- точка стала хранить `x, y, z`;
- `KD-Tree` начал делить пространство по `x`, `y`, `z` циклически;
- `nearest_neighbor` и `range_query` были переведены на 3D;
- brute force-проверки тоже были переведены на 3D.

### 3. Затем был реализован DBSCAN

Сначала была попытка ускорить `DBSCAN` через `KD-Tree`, но этот путь оказался неудачным:
- результаты начинали расходиться;
- производительность не улучшалась так, как ожидалось.

После этого архитектурное решение было изменено:
- `DBSCAN` теперь работает **по массиву точек**;
- для ускорения поиска соседей используется **пространственная сетка (`grid index`)**;
- от идеи ускорять `DBSCAN` через `KD-Tree` было решено отказаться.

Это важная часть текущей архитектуры проекта.

## Архитектура

### `src/main.c`

Главный файл программы.

Отвечает за:
- загрузку CSV в `PointArray`;
- построение `KD-Tree` по массиву точек;
- запуск режимов `-kd_insert`, `-kd_nearest`, `-kd_range`, `-dbscan`;
- brute force-проверки для `nearest` и `range`;
- замеры времени;
- сохранение результатов в CSV.

### `src/kd_tree.c` / `include/kd_tree.h`

Модуль 3D `KD-Tree`.

Реализованы:
- `insert`
- `nearest_neighbor`
- `range_query`
- `range_query_indices`
- `free_tree`

Узел дерева хранит:
- точку `Point`;
- индекс точки в исходном массиве.

### `src/dbscan.c` / `include/dbscan.h`

Модуль `DBSCAN`.

Алгоритм работает **по массиву точек**, без зависимости от `KD-Tree`.

Используется структура результата:
- `labels`
- `cluster_count`
- `noise_count`

Реализованы:
- `dbscan(...)`
- `free_dbscan_result(...)`
- `print_dbscan_summary(...)`
- `save_dbscan_result_csv(...)`

### `src/grid_index.c` / `include/grid_index.h`

Модуль пространственной сетки.

Используется для ускорения поиска соседей внутри `DBSCAN`.

Именно этот способ ускорения оказался удачным и остался в проекте как основное решение.

### `src/app_utils.c` / `include/app_utils.h`

Вспомогательный модуль.

Содержит:
- загрузку точек из CSV;
- разбор точки из строки CLI;
- brute force-функции;
- сравнение массивов точек;
- создание имён выходных файлов;
- сохранение точек в CSV.

## Сборка проекта

Для сборки используется `gcc`.

```bash
gcc -Wall -Wextra -Werror -std=c11 src/main.c src/kd_tree.c src/dbscan.c src/grid_index.c src/app_utils.c -Iinclude -lm -o robot_spatial
```

После успешной сборки появится исполняемый файл:

```bash

```

## Формат входных данных

Базовый ожидаемый формат — CSV, где каждая строка содержит одну 3D-точку:

```text
x,y,z
```

Пример:

```text
0,0,0
1,2,3
-1.5,4.2,0.8
```

### Важно

Некоторые внешние CSV-файлы содержат много служебных столбцов. В таком случае их нужно предварительно преобразовать к формату `x,y,z`.

Для набора внешних тестов использовалась команда:

```bash
awk -F, 'NR>1 {print $(NF-2)","$(NF-1)","$NF}' Tests/Still_cloud.csv > Tests/Still_xyz.csv
awk -F, 'NR>1 {print $(NF-2)","$(NF-1)","$NF}' Tests/Circle_cloud.csv > Tests/Circle_xyz.csv
awk -F, 'NR>1 {print $(NF-2)","$(NF-1)","$NF}' Tests/Random1_cloud.csv > Tests/Random1_xyz.csv
awk -F, 'NR>1 {print $(NF-2)","$(NF-1)","$NF}' Tests/Random2_cloud.csv > Tests/Random2_xyz.csv
awk -F, 'NR>1 {print $(NF-2)","$(NF-1)","$NF}' Tests/Zigzag_cloud.csv > Tests/Zigzag_xyz.csv
```

## Поддерживаемые режимы

### 1. Построение дерева

```bash
./robot_spatial <file.csv> -kd_insert
```

Пример:

```bash
./robot_spatial Tests/Still_xyz.csv -kd_insert
```

Что делает:
- загружает точки;
- строит `KD-Tree`;
- выводит дерево в консоль.

> Для больших файлов этот режим даёт слишком большой вывод, поэтому его лучше запускать либо на маленьких файлах, либо с перенаправлением в `/dev/null`.

---

### 2. Поиск ближайшего соседа

```bash
./robot_spatial <file.csv> -kd_nearest x,y,z
```

Примеры:

```bash
./robot_spatial Point_Cloud_1.csv -kd_nearest 1000.0,1000.0,1000.0
./robot_spatial Tests/Random1_xyz.csv -kd_nearest 0,0,0
```

Что делает:
- ищет ближайшую точку через `KD-Tree`;
- ищет ближайшую точку через `Brute force`;
- сравнивает результаты;
- измеряет время работы обоих способов.

---

### 3. Диапазонный поиск

```bash
./robot_spatial <file.csv> -kd_range x1,y1,z1 x2,y2,z2
```

Примеры:

```bash
./robot_spatial Point_Cloud_1.csv -kd_range 0,0,0 10,10,10
./robot_spatial Tests/Zigzag_xyz.csv -kd_range -5,-5,-5 5,5,5
```

Что делает:
- выполняет диапазонный поиск через `KD-Tree`;
- делает такой же поиск через `Brute force`;
- сравнивает количество найденных точек;
- сортирует и сравнивает результаты;
- сохраняет найденные точки в отдельный CSV-файл.

Примеры имён выходных файлов:

```text
Still_xyz_range.csv
Circle_xyz_range.csv
Random1_xyz_range.csv
```

Если файл уже существует, создаётся новый с нумерацией.

---

### 4. DBSCAN

```bash
./robot_spatial <file.csv> -dbscan <eps> <min_pts>
```

Примеры:

```bash
./robot_spatial Point_Cloud_1.csv -dbscan 1.5 2
./robot_spatial Tests/Random2_xyz.csv -dbscan 0.5 5
```

Что делает:
- запускает `DBSCAN` для 3D-точек;
- ищет соседей через `grid index`;
- выводит число кластеров;
- выводит число шумовых точек;
- измеряет время работы;
- сохраняет результат в CSV.

Формат выходного CSV:

```text
x,y,z,cluster
```

Примеры имён выходных файлов:

```text
Point_Cloud_1_dbscan.csv
Still_xyz_dbscan.csv
Random2_xyz_dbscan.csv
```

Если такой файл уже существует, создаётся новый с нумерацией.

---

### 5. Fuzzy C-means

```bash
./robot_spatial <file.csv> -cmeans ...
```

На текущем этапе алгоритм **не реализован**.

Режим распознаётся, но выводит сообщение о том, что реализация будет добавлена позже.

## Подтверждённые результаты

### `KD-Tree`: nearest neighbor

Проверено, что на всех пяти внешних файлах `KD-Tree` и `Brute force` возвращают **одинаковый ближайший сосед**.

Примеры входной точки:

```text
0,0,0
```

Во всех пяти случаях результаты совпали полностью.

На этих тестах `KD-Tree` работал быстрее `Brute force`.

### `KD-Tree`: range query

Проверено, что на всех пяти внешних файлах:
- количество найденных точек совпадает;
- после сортировки результаты `KD-Tree` и `Brute force` полностью совпадают.

Пример диапазона:

```text
-5,-5,-5 ... 5,5,5
```

Результаты:

| Файл | Найдено точек |
|------|--------------:|
| Still_xyz.csv | 1217 |
| Circle_xyz.csv | 31172 |
| Random1_xyz.csv | 85723 |
| Random2_xyz.csv | 48849 |
| Zigzag_xyz.csv | 64779 |

Это подтверждает корректность `3D range query`.

### `DBSCAN`

Для внешних тестовых файлов проверялся запуск:

```text
eps = 0.5
min_pts = 5
```

Полученные результаты:

| Файл | Точек | Кластеров | Шумовых точек | Время |
|------|------:|----------:|--------------:|------:|
| Still_xyz.csv | 2082 | 4 | 25 | 0.020083 сек |
| Circle_xyz.csv | 38341 | 9 | 88 | 2.090939 сек |
| Random1_xyz.csv | 105107 | 6 | 76 | 29.989913 сек |
| Random2_xyz.csv | 65366 | 9 | 70 | 9.696977 сек |
| Zigzag_xyz.csv | 80312 | 10 | 58 | 13.259515 сек |

Также ранее на `Point_Cloud_1.csv` был подтверждён рабочий запуск:

```bash
./robot_spatial Point_Cloud_1.csv -dbscan 1.5 2
```

Результат:
- 24 кластера;
- 1 шумовая точка;
- время около `0.83–0.89 сек`.

До перехода на `grid index` тот же этап работал заметно медленнее — около `7.8 сек`, поэтому ускорение через пространственную сетку действительно улучшило производительность.

## 2D-визуализация DBSCAN

Для построения 2D-визуализации результата `DBSCAN` используется Python-скрипт:

```bash
python3 scripts/plot_dbscan_2D.py <dbscan_result.csv>
```

Примеры:

```bash
python3 scripts/plot_dbscan_2D.py Point_Cloud_1_dbscan.csv
python3 scripts/plot_dbscan_2D.py Tests/Random1_xyz_dbscan.csv
```

Результат сохраняется в PNG-файл, например:

```text
Point_Cloud_1_dbscan_2d.png
Random1_xyz_dbscan_2d.png
```

Если `matplotlib` не установлен, можно использовать виртуальное окружение:

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install matplotlib
```

## Ограничения проекта

На текущем этапе есть несколько известных ограничений:

1. Программа ожидает входные данные в формате `x,y,z`.
2. Сложные CSV с дополнительными столбцами нужно сначала преобразовать.
3. Режим `-kd_insert` печатает дерево целиком, поэтому неудобен для больших файлов.
4. `Fuzzy C-means` пока не реализован.
5. Для `range query` `KD-Tree` по корректности работает правильно, но по времени пока не всегда выигрывает у `Brute force`.
